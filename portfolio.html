Scratch Game: DiePod

Link: DiePod

Description

The game involves a tide pod that is trying to escape being eaten
from the open mouths of teenagers. The scoring system uses the data variable
"score" that increases by 1 every 1.5 seconds. The randomness occurs in
the change of the y-position of the humans. In addition, the random changes
contribute to the difficulty of the game. The game concludes when the tide
pod touches the ground, a human, or the edge of the screen.

Reflection

I wanted to create this game to advocate for awareness of tide pod eating
and to showcase the detrimental effects of doing so. I was inspired to
show my peers through a simple game that if you eat a tide pod then it's
"GAME OVER" for life. Two successes in the development process
were creating sprites for the scoring system and drawing a tide pod. An
obstacle I had was the positioning and drawing of the humans. In the end
I drew them in costumes tab and adjusted them accordingly. Another
obstacle that I encountered was trying to have a "restart" button on the
"GAME OVER" screen, but it didn't work out and I decided to omit it altogether.
If more time was available, I would want to improve the graphics of the
humans and try to implement a "restart" button.

Algorithm

This is an algorithm because the "if" statement sets the rules for the
actions that need to be taken following a "yes" or "no" structure. This system
pictured above follows the sequence and changes the look of the tide pod
depending on whether or not the space bar is pressed.

Python Interactive Story: Twinkie Reflection

Link: Twinkie Reflection

Description

Twinkie Reflection is a "Choose Your Own Adventure" styled
game with the main character as the player. A story filled with stone cold chicken,
crocs, and yes, twinkies. With multiple endings, this story will make you think
about the decisions you choose to make.
How will your story end?

Flowchart

Development Process

For this game I took each scene of the story as individual functions,
each scene a piece for the whole story. Each function holds a specific "definition",
or the necessary code for the scene. The functions are a form of procedural
abrstraction as each function can easily be called upon with information of the specific
scene already implemented within the code, providing an efficient way to test the program
in increments. Within this process of creating multiple functions I utilized the "run"
feature in repl.it in order to check the program for any possible errors. Through repition and
focus on singular parts of the program I was able to catch grammatical errors, and I was also
able to refine the storyline through each test. At one point in a test run
a prompt was not checking for invalid user input and this was resolved by
making small adjustments to the code and running through it again, each time
working on the same part. Additionally, I was able to add some "dialouge"
to one of the prompts that the player was given if an input was invalid
and the program was set to run independently one function after the other.
